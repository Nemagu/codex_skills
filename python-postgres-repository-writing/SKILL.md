# Python Postgres Repository Writing

## Quick Start

1. Зафиксируй контракт репозитория на уровне `ports` и поведение `save/read`.
2. Определи mapping между DB-моделью и domain-объектом через `_model_to_domain`.
3. Для сохранения сразу проектируй `batch_save`, а `save` делай делегатом.
4. Разделяй создание и обновление по версии агрегата.
5. Для несеквенциальных ID (UUID и аналогов) генерируй идентификатор в коде приложения, не через БД.
6. Настрой psycopg-соединения и пулы с `row_factory=dict_row`; результаты чтения типизируй как `DictRow` и преобразуй по именам колонок.
7. Строй SQL только параметризовано, без f-строк.
8. Добавляй фильтрацию через список `conditions` и объединение `AND`.
9. Для list/count делай short-circuit: если `count == 0`, данные не запрашивай.
10. SQL-вызовы выполняй только в приватных методах; публичные методы должны делегировать.
11. В SQL запрещено бороться с конфликтами через `ON CONFLICT ... DO NOTHING/DO UPDATE`.
12. Все ошибки psycopg оборачивай в `AppInternalError`, сохраняя исходное исключение в `wrap_error`.
13. Объявляй все имена таблиц единым `StrEnum` в модуле базового Postgres-репозитория и предоставляй его каждому репозиторию через базовый класс.
14. Если репозитории одного агрегата лежат в нескольких файлах одного модуля, экспортируй их через `__init__.py` и импортируй дальше из модуля.

## Workflow

### 1. Define Contract First

- Реализуй репозиторий как адаптер к порту, не смешивай DB-детали с domain-контрактом.
- Держи публичные методы выше приватных.
- Фиксируй поведение при пустом результате, конфликте и несовпадении версии.

### 2. Centralize Mapping

- Используй единый `_model_to_domain`.
- Используй `dict_row` как стандартную row factory psycopg и `DictRow` как тип результата чтения.
- Обращайся к полям строки только по именам колонок; не используй `TupleRow`, числовые индексы и позиционные срезы.
- Не разносить преобразование по случайным helper-методам без ценности.
- Если в проекте есть унифицированный декоратор обработки доменных ошибок, применяй его к mapping.

### 3. Implement Save via Batch

- `save` вызывает `batch_save([aggregate])`.
- `batch_save` делит элементы на `to_create` и `to_update`.
- Выполняй максимум два SQL-запроса на пачку.
- Для пачек используй `executemany` внутри приватных `_batch_create/_batch_update`.
- Не скрывай конфликт в SQL-конструкциях `ON CONFLICT`: конфликт должен проявляться исключением БД.
- Для UUID-первичных ключей присваивай ID в приложении до вызова `_batch_create`.

### 4. Build Read Queries Efficiently

- Фильтры строй через список условий и `AND`-склейку.
- Считай total отдельным прямым `COUNT` без лишнего подзапроса.
- Если total равен нулю, сразу возвращай пустой список.
- Для list-запросов задавай детерминированный `ORDER BY`.
- SQL-запросы размещай в приватных методах доступа к БД.
- Используй шаблон: [read_repository_template.md](references/read_repository_template.md)

### 5. Preserve Error Semantics

- Не скрывай исключения конфликтов и нарушения ограничений.
- На границе Postgres-адаптера перехватывай ошибки psycopg и обязательно оборачивай их в `AppInternalError`.
- Передавай исходную ошибку в `wrap_error`, чтобы сохранить техническую причину и тип конфликта.
- Не заменяй ошибку пустым результатом, успешным ответом или одним логированием.
- Не делай silent retry без контрактного требования.
- Сохраняй причину ошибки и контекст операции.

### 6. Normalize Module Imports

- Если для одного агрегата есть несколько repository-классов в одном модуле (несколько файлов), экспортируй их в `module/__init__.py`.
- В вызывающем коде импортируй репозитории из модуля, а не из файлов реализации.
- Игнорируй это правило только при циклическом импорте, когда иначе собрать модуль невозможно.

### 7. Centralize Table Names

- Объявляй единый `StrEnum` со всеми фактическими именами таблиц в `base.py` или эквивалентном модуле базового Postgres-репозитория.
- Предоставляй enum всем репозиториям через атрибут базового класса, чтобы любой репозиторий мог обратиться к любой таблице.
- Используй значения enum в `Identifier`; не дублируй строковые имена таблиц и локальные `_table_name`/`_version_table_name` в конкретных репозиториях.
- Синхронизируй значения enum с именами таблиц в миграциях и проверяй это тестом.

## Anti-Patterns

- SQL через f-строки.
- SQL-запрос на каждый агрегат внутри `batch_save`.
- Генерация UUID через SQL-запрос (например `SELECT uuid...`) вместо кода приложения.
- Второй select при `count == 0`.
- Неявный mapping, размазанный по разным методам.
- Публичный метод репозитория напрямую выполняет `execute`/`executemany`.
- Подавление ошибок БД без причины.
- Маскирование ошибок SQL-запросов (перехват с логом без re-raise, замена на "пустой результат").
- Проброс ошибки psycopg через application-порт без оборачивания в `AppInternalError`.
- Использование стандартного tuple-row, `TupleRow`, числовых индексов или позиционных срезов для строк БД.
- Локальные `_table_name`, `_version_table_name` и строковые литералы имён таблиц в конкретных репозиториях.
- Использование `ON CONFLICT DO NOTHING` или `ON CONFLICT DO UPDATE` для скрытия конфликтов.
- Импорт репозиториев из файлов реализации, когда они уже должны быть реэкспортированы через `__init__.py` модуля.

## Definition of Done

- Публичные методы расположены выше приватных.
- Публичные методы не выполняют SQL напрямую и делегируют приватным методам.
- `save` делегирует в `batch_save`.
- `batch_save` не делает запрос на каждый агрегат по отдельности.
- Для UUID и других несеквенциальных ID генерация выполняется в приложении до SQL-вставки.
- SQL параметризован и не использует f-строки.
- Mapping DB-to-domain централизован и стабилен.
- Соединения используют `row_factory=dict_row`, результаты чтения типизированы как `DictRow`, mapping обращается к именам колонок.
- Фильтрация и count реализованы без лишних запросов.
- В SQL нет маскирующих конфликт конструкций `ON CONFLICT DO NOTHING/DO UPDATE`.
- Ошибки БД и конфликты оборачиваются в `AppInternalError`, а исходное исключение сохранено в `wrap_error`.
- Все имена таблиц определены единым `StrEnum` в базовом модуле и доступны каждому репозиторию через базовый класс.
- Репозитории агрегата импортируются через `module/__init__.py`, кроме редкого случая циклического импорта.
