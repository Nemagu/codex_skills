# Repository Design Checklist

## Contract and Architecture

- Репозиторий реализует порт и не тянет доменные зависимости вниз по слоям.
- Публичные методы размещены перед приватными.
- Публичные методы не выполняют SQL напрямую и делегируют в приватные методы.
- Если репозитории одного агрегата лежат в нескольких файлах одного модуля, они реэкспортируются через `__init__.py`.
- Импортировать репозитории следует из модуля, а не из файлов реализации.
- Исключение допускается только при циклическом импорте.
- Типы входа/выхода стабильны и не раскрывают DB-детали наружу.
- Поведение для empty/not-found/conflict зафиксировано в контракте.
- Все имена таблиц объявлены единым `StrEnum` в базовом модуле репозиториев.
- Enum доступен через базовый класс каждому репозиторию; локальных `_table_name` и строковых дублей нет.
- Значения enum совпадают с именами таблиц в миграциях.

## Save Semantics

- `save` вызывает `batch_save`.
- `batch_save` группирует данные на create/update.
- Для create/update используются отдельные приватные методы.
- Для пакетных методов применяется `executemany`.
- Конфликты сохранения пробрасываются выше.
- В SQL запрещено использовать `ON CONFLICT DO NOTHING/DO UPDATE` для маскирования конфликтов.
- Для UUID и других несеквенциальных ID идентификатор создается в коде приложения, без SQL-запроса к БД.

## Mapping and Errors

- Mapping-метод именован `_model_to_domain`.
- Соединение или пул настроены с `row_factory=dict_row`.
- Результаты чтения типизированы как `DictRow`, mapping использует имена колонок.
- `TupleRow`, числовые индексы и позиционные срезы строк БД не используются.
- Ошибки преобразования домена и данных обрабатываются единообразно.
- Нет лишних приватных методов, которые только переименовывают поля.

## Query Design

- Все параметры передаются через placeholders.
- Ошибки psycopg оборачиваются в `AppInternalError` с исходным исключением в `wrap_error`.
- Нет SQL через f-строки.
- Фильтры строятся как список условий с объединением `AND`.
- `COUNT` выполняется прямым запросом без лишних подзапросов.
- Есть детерминированная сортировка в list-методах.

## Read Efficiency

- При нулевом count второй запрос на данные не выполняется.
- Есть лимит/offset и контроль максимального лимита.
- В list/count не дублируется одна и та же тяжелая фильтрация без необходимости.
