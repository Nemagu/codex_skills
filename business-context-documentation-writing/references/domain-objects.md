# Доменные объекты

## Агрегат

Описывать:

- назначение и границу агрегата;
- корень и принадлежащие сущности;
- поля и бизнес-типы;
- состояния и допустимые переходы;
- инварианты;
- операции поведения и права на них;
- локальные value objects и сервисы;
- порождаемые доменные события;
- открытые вопросы.

### Структура файла агрегата

Использовать порядок разделов как базовый шаблон, удаляя неприменимые разделы:

1. `# <Название>` — назначение и граница агрегата.
2. `## Поля` — таблица `Название | Бизнес-тип | Описание`.
3. Разделы локальных value objects и доменных сервисов, если они нужны.
4. `## Состояния` и `### Допустимые переходы`, если у агрегата есть жизненный
   цикл.
5. `## Инварианты`.
6. `## Поведение` с подразделом для каждой операции агрегата.
7. `## Права доступа`, если права подтверждены как часть требований.
8. `## Доменные события` со ссылками или краткой таблицей событий.
9. `## Открытые вопросы`.

Не смешивать в одном списке инварианты, права и предусловия прикладного сценария.
Изменение состояния описывать отдельно от изменения обычных полей, если это
разные доменные операции. Не создавать пустые формальные разделы, кроме явно
сохраняемого раздела открытых вопросов.

Для каждого инварианта указывать условие, область уникальности или сравнения и момент
проверки. Различать инвариант, право доступа и правило прикладного сценария.

Считать доменный слой владельцем инвариантов: агрегат или доменный сервис должен
обеспечивать их при любом вызове доменного поведения. Если проверке нужны данные
вне агрегата, описывать необходимые сведения и их владельцев, но не переносить
сам инвариант в application-слой.

Описывать поведение агрегата отдельными методами изменения принадлежащих ему
данных. Не требовать составного метода агрегата только потому, что одна
application-команда изменяет несколько полей. Application-слой должен вызвать
необходимые доменные методы в установленном порядке и после их успешного
выполнения один раз сохранить итоговое состояние агрегата и одно событие для
полученной версии. Атомарность последовательности относить к транзакционной
границе команды.

Не описывать методы агрегата как обработчики сообщений или событий и не включать
источник вызова в их контракт. Агрегат должен предоставлять методы собственного
бизнес-поведения, изменяющие принадлежащие ему поля. Получение и разбор сообщения,
преобразование входных данных и выбор последовательности доменных методов относить
к application-операции; связь с исходным событием описывать в этой операции или
составном системном сценарии.

Если промежуточное доменное действие по требованиям образует самостоятельную
версию и событие, указывать его отдельное сохранение до следующего доменного
действия. Не объединять такие действия в одно сохранение.

Перед изменением полей агрегат должен сравнивать каждое переданное значение с
текущим. Если хотя бы одно переданное значение, включая состояние, уже установлено,
отклонять всю операцию доменной ошибкой идемпотентности: не игнорировать отдельное
поле и не применять остальные изменения частично.

Не распространять это правило автоматически на application-идемпотентность
повторного создания по внешнему событию и на принятие версий проекции. Для них
документировать отдельную подтверждённую семантику повторной обработки.

## Value object

Описывать смысл, состав значения, ограничения, нормализацию и равенство. Считать
объект неизменяемым. Локальный value object размещать в документе агрегата. Общий для
нескольких объектов одного контекста выносить в отдельный документ и перечислять
места использования.

Не вводить общий value object между контекстами только ради совпадающей структуры:
одинаковые поля не гарантируют одинакового бизнес-смысла.

## Доменный сервис

Создавать сервис только для доменного правила, которое не принадлежит естественно
одному объекту или не может быть проверено агрегатом по его внутреннему состоянию.
Документировать:

- обеспечиваемый инвариант;
- почему агрегату недостаточно собственных данных;
- необходимые входные сведения и их владельцев;
- результат проверки или вычисления;
- ожидаемые доменные отказы;
- места использования.

Не описывать в качестве доменного сервиса orchestration, сохранение, публикацию,
авторизацию или получение DTO.

## Проекция

Описывать бизнес-смысл локальной копии данных, источник, минимальный набор полей,
состояния, версию и допустимую свежесть. Проекция не должна присваивать контексту
владение данными исходного контекста.

### Структура файла проекции

Использовать порядок разделов как базовый шаблон, адаптируя его к требованиям:

1. `# <Название>` — бизнес-назначение, контекст-источник и владелец данных.
2. `## Поля` — таблица `Название | Бизнес-тип | Описание` с минимальным набором
   локально необходимых сведений.
3. `## Состояния`, если состояние источника влияет на локальные правила.
4. `## Использование` — правила и операции контекста, которым нужна проекция.
5. `## Правила обновления` — принятие версии, повторных и устаревших данных.
6. `## Согласованность` — допустимая свежесть и поведение до получения новых или
   недоступных данных.
7. `## Открытые вопросы`.

Формулировать наблюдаемое поведение проекции, не предписывая таблицу, обработчик
или транспорт. Если для повторной, устаревшей либо недопустимой версии утверждена
доменная ошибка, явно фиксировать её в документе проекции. Решение о перехвате,
повторе или успешном пропуске такой ошибки описывать на фактической границе
ответственности: в application-команде, если она перехватывает ошибку, либо в
сценарии и вызывающем адаптере, если команда пробрасывает её без преобразования.
Не приписывать обработку слою, который только передаёт исключение дальше. Явно
отделять владение исходными данными от владения их локальным представлением.
