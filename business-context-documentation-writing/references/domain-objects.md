# Доменные объекты

## Агрегат

Описывать:

- назначение и границу агрегата;
- корень и принадлежащие сущности;
- поля и бизнес-типы;
- состояния и допустимые переходы;
- инварианты;
- операции поведения и права на них;
- локальные value objects и сервисы;
- порождаемые доменные события;
- открытые вопросы.

Для каждого инварианта указывать условие, область уникальности или сравнения и момент
проверки. Различать инвариант, право доступа и правило прикладного сценария.

## Value object

Описывать смысл, состав значения, ограничения, нормализацию и равенство. Считать
объект неизменяемым. Локальный value object размещать в документе агрегата. Общий для
нескольких объектов одного контекста выносить в отдельный документ и перечислять
места использования.

Не вводить общий value object между контекстами только ради совпадающей структуры:
одинаковые поля не гарантируют одинакового бизнес-смысла.

## Доменный сервис

Создавать сервис только для доменного правила, которое не принадлежит естественно
одному объекту или не может быть проверено агрегатом по его внутреннему состоянию.
Документировать:

- обеспечиваемый инвариант;
- почему агрегату недостаточно собственных данных;
- необходимые входные сведения и их владельцев;
- результат проверки или вычисления;
- ожидаемые доменные отказы;
- места использования.

Не описывать в качестве доменного сервиса orchestration, сохранение, публикацию,
авторизацию или получение DTO.

## Проекция

Описывать бизнес-смысл локальной копии данных, источник, минимальный набор полей,
состояния, версию и допустимую свежесть. Проекция не должна присваивать контексту
владение данными исходного контекста.
