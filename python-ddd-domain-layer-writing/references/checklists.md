# Checklists

## Перед началом — вопросы самопроверки

Перед тем как создавать сущность, ответь на четыре вопроса:

1. **Это агрегат или проекция?**
   - Источник правды для сущности **в этом сервисе** → агрегат.
   - Источник правды **в другом сервисе/BC**, у нас только локальное представление → проекция.

2. **Сколько у сущности состояний?**
   - Два (`ACTIVE` / `DELETED`) → стандартный `State`, наследование `EntityWithState` / `ProjectionWithState`.
   - Больше → собственный enum (`<Name>State`), наследование `Entity` / `Projection` напрямую.

3. **Нужны ли domain-сервисы и repository-интерфейс?**
   - Бизнес-правилу требуется чтение → да, `service.py` + узкий `repository.py`.
   - Только владельческая политика без I/O → только `service.py` с Policy сервисом, `repository.py` не нужен.
   - Ничего из этого → ни `service.py`, ни `repository.py`.

4. **Какие связи с другими агрегатами / проекциями?**
   - Если связь 1:1 с внешней сущностью (для проекции) — заранее зафиксируй ID-parity с возможным локальным агрегатом.
   - Если хранятся ссылки — это набор ID-VO, а не объекты целиком.

Если хотя бы на один вопрос не ответил — план не готов, согласовывай его с пользователем.

## Чек-лист добавления агрегата

### Шаг 1 — Создать поддиректорию

`domain/<aggregate_name>/` (snake_case, единственное число).

### Шаг 2 — `value_object.py`

Создать в порядке зависимости:

- [ ] `<Aggregate>ID` (Identity VO).
- [ ] Все Validated VO агрегата.
- [ ] Все Enum VO агрегата.
- [ ] Multi-field VO, если есть.
- [ ] Если расширенное состояние — `<Aggregate>State` enum-VO здесь же.

### Шаг 3 — `entity.py`

- [ ] Импорты: `EntityWithState` или `Entity` из `domain.entity`, нужные ошибки, ID-VO связанных агрегатов из их витрин, локальные VO из `value_object.py`, общие VO из `domain.value_object`.
- [ ] Класс `<Aggregate>(EntityWithState)` или `<Aggregate>(Entity)`.
- [ ] `_DELETED_MSG: ClassVar[str] = ...` (плюс другие `_*_MSG` при расширенном состоянии).
- [ ] Конструктор: параметры — VO; вызов `super().__init__(...)` с `aggregate_name=AggregateName("...")`; присваивание приватных полей.
- [ ] Properties для всех приватных полей (read-only, имя без `_`).
- [ ] Коллекции VO хранятся и возвращаются как `frozenset` или `tuple`.
- [ ] Методы поведения по каноничному шаблону: `_check_state()` → идемпотентность → мутация → `_update_version()`.
- [ ] При расширенном состоянии — собственные `activate`/`delete`/`new_state`/доп. переходы и собственный `_check_state`.
- [ ] Реализация `_error_data`.
- [ ] Если есть кросс-агрегатные инварианты — публичные `validate_*` методы.
- [ ] Если участвует в политиках — `raise_*` методы (бросают `EntityPolicyError`).

### Шаг 4 — `factory.py`

- [ ] Класс `<Aggregate>Factory`, все методы `@staticmethod`.
- [ ] `new(...)`: принимает ID и бизнес-данные только как VO; стартовые `state`, `version` и business-default поля задаёт сама фабрика.
- [ ] `restore(...)`: принимает все поля только как VO, включая `state` и `version`; возвращает агрегат в любом валидном состоянии.
- [ ] Raw-данные преобразуются в VO до вызова фабрики; методов `from_raw` нет.

### Шаг 5 — `repository.py` (опционально)

Создаётся, только если есть domain-сервис, которому нужно чтение из БД.

- [ ] Класс `<Aggregate>ReadRepository(ABC)`.
- [ ] Все методы — `async`, `@abstractmethod`, тело `...`, имя `by_<criteria>`.
- [ ] Параметры — VO, возврат — `<Aggregate> | None` или `list[<Aggregate>]`.

### Шаг 6 — `service.py` (опционально)

- [ ] Каждый сервис проверяет одно конкретное бизнес-правило и не создаёт агрегат.
- [ ] Имя сервиса отражает правило, а не техническую категорию.
- [ ] Repository-интерфейс добавлен только если он требуется этой проверке.

### Шаг 7 — `__init__.py`

- [ ] Импорт всех публичных классов агрегата.
- [ ] `__all__` со всеми именами в алфавитном порядке.
- [ ] Импорты внутри файла отсортированы по полному пути.

## Чек-лист добавления проекции

Отличается от агрегата в нескольких местах. Шаги, где нет различий, — кратко.

### Шаг 1 — Поддиректория

`domain/<projection_name>/`.

### Шаг 2 — `value_object.py`

- [ ] `<Projection>ID`. **Если ID-parity с внешней сущностью или связанным агрегатом — UUID должен совпадать**; зафиксируй это в комментарии или в архитектурных заметках.
- [ ] Validated/Enum VO проекции.
- [ ] При расширенном состоянии — `<Projection>State`.

### Шаг 3 — `projection.py` (не `entity.py`!)

- [ ] Импорты: `ProjectionWithState` или `Projection` из `domain.projection`.
- [ ] Класс `<Projection>(ProjectionWithState)` или `<Projection>(Projection)`.
- [ ] **Никаких** `_DELETED_MSG` — у проекции нет `_check_state`.
- [ ] Конструктор с `super().__init__(..., projection_name=ProjectionName("..."))`.
- [ ] Properties для всех приватных полей.
- [ ] Методы применения событий (`new_state` и т.п.) по шаблону: идемпотентность → мутация. **Без `_check_state` и без `_update_version`.**
- [ ] При расширенном состоянии — собственный `new_state(state)` и доп. методы под нужные события (часто хватает одного `new_state`).
- [ ] Реализация `_error_data` с `subject=self._projection_name.name`.

### Шаг 4 — `factory.py`

- [ ] **Один метод** `restore(...)` — потому что у проекции нет семантической разницы между `new` и `restore`. Все параметры — VO, включая `state` и `version`.

### Шаг 5 — `repository.py` (обычно есть)

- [ ] Read-репозиторий для domain-сервисов и для синхронизации.

### Шаг 6 — `service.py` (опционально)

- [ ] Чаще всего у проекций есть `<Projection>UniquenessService` (валидация существования при создании связанного агрегата).
- [ ] Сервис не создаёт проекцию, а проверяет отдельное бизнес-правило.
- [ ] `Policy` — почти не встречается у проекций.

### Шаг 7 — `__init__.py`

- [ ] Импорты всех публичных классов проекции.
- [ ] `__all__` алфавитно.

## Что делать после domain-слоя

Этот скил отвечает только за domain. После добавления агрегата/проекции для полной интеграции потребуется:

- **`application/ports/repositories/<aggregate>.py`** — полный repository-интерфейс с read + save + version.
- **`application/commands/...` и/или `application/queries/...`** — use case'ы, использующие новый агрегат.
- **`infrastructure/db/...`** — реализация репозиториев, миграции.
- **`presentation/api/...`** — роутер и Pydantic-модели, если агрегат экспонируется по API.

Эти шаги — за пределами domain-скила.

## Финальная проверка

Перед тем как считать domain-часть готовой:

- [ ] Все импорты внутри `domain/` идут только из `domain/` или стандартной библиотеки.
- [ ] Витрина (`__init__.py`) обновлена, `__all__` синхронизирован с реальными именами.
- [ ] Каждое мутирующее поведение агрегата защищено `_check_state` (где применимо) и проверкой идемпотентности.
- [ ] Каждое поведение, меняющее состояние агрегата, заканчивается `_update_version`.
- [ ] У проекции **нет** `_update_version` в state-методах и нет `_check_state`.
- [ ] Все доменные ошибки бросаются с kwargs (`msg=`, `subject=`, `data=`).
- [ ] `data` ошибок собирается по конвенции (английский snake_case ключ-сущность на верхнем уровне, поля сущности внутри).
- [ ] Все приватные поля начинаются с `_` и имеют read-only `@property`.
- [ ] VO иммутабельны (`@dataclass(frozen=True)` или `StrEnum`).
