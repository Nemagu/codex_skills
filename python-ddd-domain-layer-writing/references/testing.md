# Тестирование domain-слоя

## Обязательные проверки

- Размещай unit-тесты в `src/tests/unit/domain/` по структуре product-кода.
- Поддерживай 100% покрытия domain-слоя, но проверяй поведение и инварианты, а не строки ради метрики.
- Для VO проверяй нормализацию, допустимые границы и каждую ветку доменной ошибки.
- Для агрегата проверяй начальное состояние, допустимые переходы, идемпотентность, запрещённые операции и error data.
- После успешной мутации проверяй новое состояние и версию.
- После ошибки проверяй, что состояние и версия агрегата не изменились.
- Для фабрики отдельно проверяй `new()` со стартовыми значениями и `restore()` без изменения переданных VO.
- Для domain-сервиса проверяй успешное правило и каждую доменную ошибку через упрощённую реализацию repository-интерфейса.

## Фикстуры и фабрики

- Создавай тестовые агрегаты через factory-функцию, вложенную в pytest fixture.
- Значения по умолчанию должны создавать валидный объект; изменяй только значимые для сценария поля.
- При повторном использовании вынеси fixture в ближайший общий `conftest.py`.
- Перед созданием новой общей фабрики проверь, что аналогичной нет в других domain-тестах.
- Не дублируй fixtures и не скрывай случайность, если она не является предметом теста.

## Структура сценариев

- Один тест проверяет один поведенческий контракт.
- Объединяй одинаково устроенные переходы и ошибки через `pytest.mark.parametrize` с обязательными `ids`.
- Проверяй публичное поведение, не редактируй приватные поля агрегата.
- Запускай точечные domain-тесты во время работы, затем полный `uv run pytest` и отчёт покрытия.
