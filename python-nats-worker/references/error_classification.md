# Error Classification

Матрица «исключение → ACK / NAK / TERM» для `NatsConsumerWorker._resolve_message_response_type` и объяснение, почему каждое решение именно такое.

## Базовые понятия NATS / JetStream

| Метод | Что делает в JetStream |
|---|---|
| `msg.ack()` | Сообщение успешно обработано, **больше не доставляется**. |
| `msg.nak()` | Сообщение **временно отвергнуто**; JetStream доставит повторно (учитывая `max_deliver` стрима). |
| `msg.term()` | Сообщение **навсегда отвергнуто**; повторных доставок не будет. По достижении `max_deliver` происходит то же самое автоматически. |

**DLQ в JetStream** — это не отдельный subject и не отдельная публикация. JetStream сам считает счётчик доставок (`AckPendingDelivery`); по превышении `max_deliver` сообщение либо просто отбрасывается, либо публикуется в advisory-subject (зависит от настройки стрима). `term()` — это форсированное «сразу в DLQ» вместо ожидания `max_deliver` повторов.

## Матрица

| Исключение | Ответ | Почему |
|---|---|---|
| `None` (handler завершился без исключения) | `ACK` | Use case успешно отработал. |
| `JSONDecodeError` | `TERM` | Битый JSON — повторная доставка не починит. |
| `_InvalidPayloadError` | `TERM` | Pydantic не валидировал payload против схемы. Повторный приём не поможет: payload не изменится. |
| `AppNotFoundError` | `NAK` | Связанная сущность ещё не создана — типично для гонок между consumer-ом и upstream-сервисом. Скорее всего через секунду она появится. |
| `AppInternalError` | `NAK` | Внутренний сбой (БД упала, побочный сервис недоступен). Ретрай вероятно поможет. |
| `DomainError` (любой подтип кроме `EntityPolicyError` — он не возникает в consumer-е) | `TERM` | Доменная ошибка — нарушение инварианта; повторная доставка тот же payload снова к тому же инварианту приведёт. |
| `AppError` (прочее, не покрытое более конкретными ветками) | `TERM` | Безопасный default для прикладных ошибок: лучше отвергнуть, чем зациклить retry. |
| Любое другое исключение | `NAK` | Неизвестная ошибка — могла быть транзиентной. Доставим повторно, если повторится — `max_deliver` стрима остановит. |

Порядок проверок в `_resolve_message_response_type` **важен** — более специфические `isinstance`-ветки идут раньше, потому что `_InvalidPayloadError` наследует `AppInternalError`, а `AppNotFoundError` тоже `AppError`. Иначе они попадут не в ту ветку.

## NAK vs TERM — главное правило

Спросить «повторный приём через 5 секунд имеет шанс пройти успешно?»:

- **Да** → `NAK` (БД упала, related-агрегат ещё не приехал, сетевой таймаут).
- **Нет** → `TERM` (битый payload, нарушение инварианта, неподдерживаемая версия схемы).

Главный антипаттерн — TERM-ить временные ошибки. Сообщение пропадает, восстановления нет.

Второй антипаттерн — NAK-ать битый payload. JetStream упорно гоняет его до `max_deliver`, забивая consumer; за это время реальные сообщения копятся.

## Где живёт `_InvalidPayloadError`

`_InvalidPayloadError` — приватный класс в `consumer.py`, наследник `AppInternalError`. Используется **только** для оборачивания `ValidationError` из pydantic при распаковке payload.

Почему наследник `AppInternalError`, а не самостоятельный класс:

- Чтобы попасть в общий log-формат и иметь `msg`/`action`/`data`-поля как у других прикладных ошибок.
- Чтобы случайный `isinstance(error, AppError)` где-то выше не классифицировал его как TERM-by-default через ветку `AppError`. Своя ветка `(_InvalidPayloadError, JSONDecodeError) → TERM` стоит **раньше** ветки `AppInternalError → NAK` именно по этой причине: для `_InvalidPayloadError` нужен TERM (битый payload), а не NAK (как для родителя).

Приватность (префикс `_`) фиксирует: класс не для импорта в use case-ы; он живёт строго в transport-слое.

## Что делать с unhandled `BaseException`

Default-ветка матрицы — `NAK`. Логика: если воркер не знает, что это была за ошибка, безопаснее отдать на повтор и дать JetStream-у сделать своё (через `max_deliver`), чем потерять сообщение. Если такие unhandled ошибки регулярно появляются — фикс в коде: добавь конкретную ветку с правильным response_type и логируй так, чтобы можно было отследить.

## `_log_processing_error` и log-levels

Логирование ортогонально классификации, но идёт рядом по той же таблице типов:

| Исключение | Уровень |
|---|---|
| `DomainError` | `warning` (ожидаемая ошибка домена) |
| `_InvalidPayloadError`, `JSONDecodeError` | `error` (битый payload — повод для алерта) |
| `AppInternalError` | `error` (нужен фикс инфраструктуры/кода) |
| `AppError` (прочее) | `warning` |
| Неклассифицированное | `error` |

Префикс `[<name>]` (имя подписки из `_create_tasks`) ставится во все лог-сообщения, чтобы по логу было видно, какая подписка генерит шум.

## Tuning `max_deliver` на стриме

`max_deliver` задаётся в `StreamConfig` (или per-consumer) — не в воркере. Рекомендация:

- Слишком малое значение (например, 1) → любая транзиентная ошибка теряет сообщение.
- Слишком большое → битый payload через NAK будет крутиться очень долго (но классификатор должен дать ему TERM, так что это страховка).
- Разумный baseline: 5–10. Конкретное значение — продуктовое решение, на стороне `infrastructure/config/nats.py` или конфига стрима в инфре.
