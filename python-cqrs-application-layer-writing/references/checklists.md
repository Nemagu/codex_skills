# Checklists

Пути ниже используют предпочтительную форму единственного числа. Если проект последовательно использует множественное число, сначала согласуй сохранение конвенции или отдельное переименование.

## Command

1. Подтверди, что сценарий изменяет состояние или инициирует прикладной побочный эффект.
2. Создай `application/command/{public,private}/<aggregate>/<action>.py`.
3. Определи неизменяемый `@dataclass(slots=True, frozen=True)` с примитивами и stdlib-типами.
4. Выбери зависимости: UoW для согласованных изменений, прямой порт для одной атомарной операции.
5. До транзакции преобразуй вход в VO, если для этого не нужен I/O.
6. Для public-сценария проверь существование, состояние, роль инициатора и доменную policy над целевым агрегатом.
7. Выполни доменный мутатор.
8. В одном UoW сохрани все согласованные изменения текущего состояния, версии и outbox-намерения.
9. Верни выходной DTO или `None`; не возвращай domain-объект.
10. Обнови агрегатную витрину `__init__.py`.

## Query

1. Подтверди отсутствие изменения состояния; read-only порт не делает сценарий Command.
2. Создай `application/query/<aggregate>/<action>.py` и назови вход `<Get|List>...Query`.
3. Определи неизменяемый вход с примитивами и stdlib-типами.
4. Инжектируй специализированный read-порт напрямую, если транзакция не нужна.
5. Проверь прикладную авторизацию, если сценарий public.
6. Верни выходной DTO или `tuple[list[DTO], int]` для пагинированного списка.
7. Обнови агрегатную витрину `__init__.py`.

## Repository-порты агрегата

1. Создай `application/port/repository/<aggregate>.py`.
2. Расширь domain repository только операциями, нужными use case-ам.
3. Добавь version/outbox/subscription-интерфейсы только при фактической потребности.
4. Объяви `<Aggregate>Repositories` рядом с интерфейсами этого агрегата как `@dataclass(slots=True)` без `frozen=True`.
5. В `application/port/repository/__init__.py` добавь только re-export и алфавитный `__all__`.
6. Добавь repository-группу в `UnitOfWork` через абстрактное property.
7. Если DTO используют несколько портов/адаптеров, вынеси его в `application/port/dto/<aggregate>.py`.

## DTO

1. Определи роль: входной Command/Query, выходной DTO или внутренний port DTO.
2. Назови тип по смыслу данных (`UserDTO`, `UserVersionDTO`, `UserEventDTO`, `ClaimsDTO`).
3. Не используй `Simple/Detail/Full/Short` без согласованной системы представлений.
4. Для входных и выходных DTO используй только примитивы, stdlib и допустимую композицию DTO.
5. Domain entity и VO разрешены только во внутренних port DTO.
6. Оформи снимок как `@dataclass(slots=True, frozen=True)`.
7. Не добавляй I/O, бизнес-поведение и сериализацию.

## Publisher use case

1. Прочитай неопубликованный batch без транзакции, если чтение не требует согласованного изменения.
2. При пустом batch заверши сценарий без UoW и вызова publisher.
3. Передай весь batch в универсальный `EventPublisher.batch_publish`.
4. При ошибке не открывай UoW отметки и не подавляй исключение.
5. После полного успеха открой короткий UoW и отметь весь batch опубликованным.
6. При частичном физическом успехе считай неуспешным весь batch; полагайся на стабильный event id и идемпотентную обработку повторов.
7. Оставь сериализацию, выбор payload и subject в infrastructure.

## Application-ошибка

1. Не лови ошибку в use case.
2. В infrastructure-адаптере преобразуй техническую ошибку в подходящий application-тип.
3. Сразу заполни стабильные `msg` и `action` без f-строки с исходным исключением.
4. Сохрани техническую причину в `wrap_error`.
5. Не возвращай `AppError | None` как результат порта: порт возвращает успешное значение или бросает исключение.
6. Не добавляй подкласс на агрегат; расширяй иерархию только для новой прикладной семантики.
